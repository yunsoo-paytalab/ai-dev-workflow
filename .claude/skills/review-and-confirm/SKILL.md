# Review & Confirm Skill

사용자 검토 및 승인 프로세스를 표준화하는 스킬입니다.

## 사용 시점

워크플로우에서 사용자 검토가 필요한 모든 Phase에서 사용

## 표준 선택지

### 기본형 (승인/직접입력)

| 선택         | 처리                              |
| ------------ | --------------------------------- |
| ✅ 승인      | 다음 Phase로 진행                 |
| 💬 직접 입력 | 요청 수행 후 **선택지 다시 표시** |

### 커밋형 (Commit 선택지)

| 선택                | 처리                              |
| ------------------- | --------------------------------- |
| ✅ Commit하고 다음  | git commit 후 진행                |
| ⏭️ Commit 없이 다음 | git add만 하고 진행               |
| 💬 직접 입력        | 요청 수행 후 **선택지 다시 표시** |

### 다중 선택형 (상황별)

| 선택         | 처리                              |
| ------------ | --------------------------------- |
| ✅ 옵션 A    | 옵션 A 실행                       |
| ✅ 옵션 B    | 옵션 B 실행                       |
| 💬 직접 입력 | 요청 수행 후 **선택지 다시 표시** |

## 핵심 규칙

> 🚨 **CRITICAL - 반드시 준수**
>
> 1. **명확한 선택지 선택 전까지 절대 다음 Phase로 진행 금지**
> 2. **직접 입력 처리 플로우:**
>    - 사용자 요청 수행
>    - 결과 공유
>    - **동일한 선택지 다시 표시**
> 3. **이 사이클은 사용자가 명확한 선택지를 선택할 때까지 무한 반복**

## 사용 방법

워크플로우 커맨드에서 다음과 같이 참조:

```markdown
### Phase N: Review & Confirm

> 💡 **Skill**: `@.claude/skills/review-and-confirm/SKILL.md`

검토 항목:

- 항목 1
- 항목 2
```

## 직접 입력 처리 예시

```
사용자: "API 응답 형식을 수정해줘"
     ↓
Claude: 요청 수행 (API 응답 형식 수정)
     ↓
Claude: 결과 공유 ("API 응답 형식을 다음과 같이 수정했습니다...")
     ↓
Claude: **선택지 다시 표시** (✅ 승인 / 💬 직접 입력)
     ↓
사용자: "✅ 승인"
     ↓
Claude: 다음 Phase로 진행
```

## 주의사항

- 직접 입력 시 절대 자동으로 승인 처리하지 말 것
- 사용자가 "좋아", "오케이", "진행해" 등 비공식 승인 표현을 해도 선택지 다시 제공
- 반드시 선택지에서 **명시적으로 선택**해야만 진행
