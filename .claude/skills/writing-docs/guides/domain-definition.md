# Domain Definition 문서 작성 가이드

## 핵심 원칙

**"분석 결과를 바탕으로 도메인을 설계하고 구조화된 문서로 작성"**

## 분석 → 설계 매핑

| domain-research.md (분석) | domain-definition.md (설계)  |
| ------------------------- | ---------------------------- |
| 요구사항 요약             | 개요 / 비즈니스 목표         |
| 비즈니스 도메인 파악      | → **Bounded Contexts 설계**  |
| 관련 기존 코드            | → **엔티티/인터페이스 설계** |
| 기술적 제약사항           | 기술적 제약사항              |
| 추가 조사 필요 사항       | 미결 사항                    |

## 구현 상태 필터링

⚠️ **중요: 신규 개발과 관련된 도메인/엔티티에만 집중**

`domain-research.md`의 구현 상태(Section 1.2)를 기반으로:

**domain-definition에 상세히 포함할 것:**
- `신규`: 구현되지 않은 기능과 관련된 도메인/엔티티
- `부분`: 추가 작업이 필요한 기능과 관련된 도메인/엔티티
- `수정`: 수정이 필요한 기능과 관련된 도메인/엔티티

**domain-definition에서 최소화할 것:**
- `완료`: 이미 구현 완료된 기능의 도메인/엔티티
  - 신규 개발 항목이 의존하는 경우에만 간단히 언급
  - 상세 정의는 생략

**이유:**

domain-definition은 앞으로 개발할 기능의 설계 문서입니다. 이미 완료된 도메인까지 상세히 작성하면:
- 문서가 불필요하게 길어짐
- 실제 개발 범위 파악이 어려워짐
- 토큰 낭비 및 가독성 저하

## 설계 항목

이 에이전트에서 설계해야 할 항목:

- **Bounded Contexts**: 비즈니스 도메인 파악 결과를 바탕으로 **신규 개발 관련** 컨텍스트 경계 설계
- **엔티티/인터페이스**: 관련 기존 코드를 참고하여 **신규/수정 필요 엔티티** 설계 (프로젝트 언어에 맞게 코드 블럭으로 작성)
- **비즈니스 규칙**: 요구사항과 도메인에서 도출 (**신규 개발 관련**)
- **컨텍스트 관계도**: 도메인 간 관계를 시각화 (신규 개발 항목 중심)
- **도메인 용어집**: 분석 과정에서 발견된 **신규 용어** 정리

## 클라이언트 인터페이스 규칙

- **포함**: 클라이언트에서 사용하는 도메인 엔티티, 타입
- **제외**: 백엔드 DTO (Request/Response 인터페이스)
- DTO 변환 레이어가 있는 경우 언급

## 작성 원칙

### 비즈니스 중심

- 기술보다 비즈니스 요구사항에 집중
- "어떻게"보다 "무엇을"에 집중

### 명확성

- 모호한 표현 배제
- 구체적인 예시 제공
- 용어 정의 명확히

### 구조화

- 계층적 구조 유지
- 일관된 포맷 사용
- 참조 가능한 ID/이름 부여
